Hypertext Transfer Protocol（ HTTP ）

~HTTPを定義する各種~文書

~HTTPは、分散型の協調的な~hypertext情報~system用の，~statelessな応用~levelの~protocolである。 ◎ The Hypertext Transfer Protocol (HTTP) is a stateless application-level protocol for distributed, collaborative, hypertext information systems.

~HTTPは、以下に挙げる［ IETF ~RFC, IANA 登記簿 ］により定義される。 ◎ HTTP is defined by these IETF RFCs and IANA registries.

【 以下における各項の先頭の “RFC…” に付与された~linkは，原文を指し、後続の~textに付与された~linkは，（もしあれば）日本語訳を指す。 一部の仕様には、便利~用に，外部~siteによる［ 日本語訳（ “外部~site” と付記される）／日本語情報（ “参考情報” と記される） ］も挙げている。 】

中核~仕様

~HTTP~protocolの “中核” を成す意味論は、次の仕様により定義される： ◎ The “core” semantics of the HTTP protocol are defined by this specification:

  • RFC7231： ~HTTPの意味論と内容 【意味論は、 RFC 7230 の一部, 主に `RFC7230/2$sec においても定義される。】 ◎ RFC7231: HTTP Semantics and Content

そのような意味論を “伝送路~上” に表出する仕方には、次の 2 つがある： ◎ Those semantics are expressed “on the wire” in two different ways:

  • RFC7230： ~HTTP11 ◎ RFC7230: HTTP/1.1
  • RFC7540： ~HTTP2（外部~site） ◎ RFC7540: HTTP/2

~HTTP2は、次も利用する： ◎ HTTP/2 also uses:

  • RFC7541： HPACK: ~HTTP2用の~header圧縮（外部~site） ◎ RFC7541: HPACK Header Compression for HTTP/2

次に挙げる~HTTP機能を実装するのは任意選択~であるが、広範に利用されている： ◎ These HTTP functions are optional to implement, but widely used:

  • RFC7232： ~HTTP条件付き要請 ◎ RFC7232: HTTP Conditional Requests
  • RFC7233： ~HTTP範囲~要請 ◎ RFC7233: HTTP Range Requests
  • RFC7234： ~HTTP~caching ◎ RFC7234: HTTP Caching
  • RFC7235： ~HTTP認証 ◎ RFC7235: HTTP Authentication

これらの~RFCは、総集的に，~HTTPを定義している先行する~RFCすべてを — RFC1945, RFC2068, RFC2616, RFC2617 を含め — 廃用にする。 ◎ These RFCs collectively obsolete all preceding RFCs defining HTTP, including RFC1945, RFC2068, RFC2616 and RFC2617.

中核~仕様は、現在， 改訂~下にある — 課題を見出された方は、そこへ~~提起されたし。 【改訂~下にある仕様, それらの和訳を次に挙げる：】 ◎ The core specifications are currently under revision. If you find issues, please bring them up there!

  • Semantics： 意味論（および， 要請~methodと要請~header／ 応答~状態s~codeと応答~header ） — これは、中核~仕様（ Messaging, Caching が受持つ部分は除く）, および一部の~HTTP拡張に取って代わることになる。
  • Caching： ~cache法 — これは、 RFC7234 に取って代わることになる。
  • Messaging： ~message法 — これは、 RFC7230 の大部分（~HTTP11に特有な部分）に取って代わることになる。

拡張~登記簿

~HTTPには、［ ~method, 状態s~code, ~header ］を含め，多数の拡張がある。 それらはすべて — 適切な登記簿に挙げられる限り — ~HTTPの “一部を成す” ことに注意： ◎ There are a large number of HTTP extensions, including methods, status codes and headers. It’s important to note that they are all “part of” HTTP, as long as they’re listed in the appropriate registries:

  • ~HTTP~method登記簿 — 詳細情報 ◎ HTTP Method Registry — more info
  • ~HTTP状態s~code登記簿 — 詳細情報 ◎ HTTP Status Code Registry — more info
  • ~protocolが `http^c の下での ~message~header登記簿 — 詳細情報 ◎ Message Header Registry where protocol is ‘http’ — more info
  • ~HTTP2の各種~parameter — ~frame種別, 設定群, ~error~code ◎ HTTP/2 Parameters — Frame types, settings, error codes

個々の~headerには、許可-可能な値を規制する，自前の登記簿がある — 次に挙げるものを含め： ◎ Individual headers have their own registries that regulate permissible values, including:

  • ~HTTP認証~scheme登記簿 （ `HTTP Authentication Scheme Registry^en ） — 詳細情報 ◎ HTTP Authentication Scheme Registry — more info
  • ~HTTP~cache指令~登記簿 （ `HTTP Cache Directive Registry^en ） — 詳細情報 ◎ HTTP Cache Directive Registry — more info
  • ~HTTPの各種~parameter — 内容~符号法, 転送~符号法, `Forwarded^h ~header, `Prefer^h ~header, `Range^h に関係する~header ◎ HTTP Parameters — Content-codings, transfer-codings, Forwarded, Prefer, and Range-related headers

日本語への翻訳に関する注記

RFC 原文は，ほぼ素のテキストであり、この日本語訳では、 HTML マークアップ ／ CSS スタイル の利用により，記述が等価になる所では、一部の対訳を省略している。 例えば：

  • 用語の定義箇所を示す，節 番号への参照の多くは、代わりに用語そのものにリンクをあてがった上で，節 番号の記述を省略している。
  • リンク先の説明で代用になるときには、記述を省くことがある。
  • 一部の語 — 例えば “状態°コード（ status code ）” は、コード番号が記されている所では、（その番号に専用のスタイルがあてがわれるため，）自明なので省略することもある。

また、見通しを良くするため、原文のかなりの割合の段落は，マークアップを利用してより明示的な文脈階層に細分化している。 例えば：

  • 構造を細分化すると同時に，複数の文から共有されている語句（例えば主語）, あるいは文脈を指示している記述を一箇所に集約している所もある。
  • 同じ文脈に属する複数の段落を，リスト化などにより一つのブロック階層にまとめている所もある。
  • 文や語句の互いの論理的なつながりや関係性［ を文脈から推し量る必要がある ／ がそのまま日本語訳すると不明瞭になる ］所では、階層化により明確化している。

原文には、（特に同じ段落内で，）同じ対象を，異なる［ 言い回し, または語句 ］により言及している箇所も多数見られる。 紛らわしいので、細分化に際しては，そのような言及もなるべく一つに［ 集約, あるいは結合 ］している（例えば、 明示的な指示代名詞マークアップ などを用いて — 日本語には “the” が無い）。

以上に述べたようなマークアップは，訳者による解釈も含むので、解釈に誤りがあれば，それを反映することになる（例えば、原文の語句（例えば “host” ）が、構文の構成子か, 用語か, 意味の記述か, いずれを表すのか、あるいはそれらを兼ねているとしても（多くの場合，そうである），いずれが主として 意図されている／念頭に置かれているかは、必ずしも自明ではない）。

一部の語の対訳は，“忠実さに欠く” 語, あるいは一般に見受けられる語と異なる語に代えている。 例えば：

  • 原文の語 “header field” は、一律に “ヘッダ（ header ）” と略して対訳している。 “header field” の略記として用いられている語 “field” も同様に置換している。 “ヘッダ” が他の意味で用いられる所は無く， “フィールド” より “ヘッダ” の方が一般に通りがよいと見受けられるので。
  • 原文の語 “media type” （媒体型, Internet media type の略称）は、 RFC 以外の Web 標準に共通して利用されている用語 “MIME 型” に一律に置換している。
  • “回送（ forward ）” ： “転送” あるいは “自動転送” という対訳もあるが、前者は “transfer” の対訳と衝突し，後者は長い（また， “automatic な transfer” と誤解され易くもなる）ので。
  • “条態（ condition ）” ： これは、訳者による造語（条件＋状態）である。 “状況／状態” , あるいは “条件” と訳されることが多いが、前者は “situation”, “circumstance” ／ “state” などの対訳と衝突し，後者は文脈にそぐわないことが多々あるので（無論、 “コンディション” とも訳せるが，用語 “条件付き（ conditional ）” との関連性もあるので）。
  • “全部的（ full ）” — “完全（ complete ）” との区別を要するため。 また、 “部分的（ partial ）” との対比も兼ねるため。
  • “検証°（ verify ）” （正しく作られたかどうかを検証）— “検証（ validate ）” （作られたものが正しいかどうかを検証）と区別するため。
  • “状態°（ status ）” （状態の集合／状態を［指示する／識別する］もの） — “状態（ state ）” （ある特定の状態）と区別するため。
